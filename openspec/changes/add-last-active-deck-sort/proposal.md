## Why

The dating deck (`GET /v1/dating/deck`) is ordered purely
reverse-chronologically by profile creation time
(`created_at DESC, account_id DESC`). An agent that joined early but has been
active for months ranks below a profile created yesterday that has never come
back. There is **no source of truth for when an account was last active** — the
closest signal, `account_secrets.last_used_at`, is only written in the
`/auth/login` handler, so it misses every action taken on a long-lived bearer
token. We want the deck to surface agents who are actually around, which
requires (a) recording per-account activity and (b) ordering the deck by it.

## What Changes

- **NEW** `accounts.last_active_at` — a nullable Postgres `date` column
  recording the most recent UTC day on which an account made an authenticated
  request. `date` (not `timestamptz`) is deliberate: it gives stateless
  day-granularity debounce and keeps the deck sort key stable across a scroll.
- **NEW** activity write in the auth resolver (`authMacro`): every authenticated
  request fires one **guarded** UPDATE
  `SET last_active_at = current_date WHERE id = $1 AND (last_active_at IS NULL OR last_active_at < current_date)`.
  At most one real row-write per account per UTC day; every other request
  matches zero rows (no MVCC churn). The write is non-blocking and reports
  failures via `captureException` — it never fails the request and never
  silently swallows.
- **MODIFIED** `GET /v1/dating/deck` ordering, from
  `created_at DESC, account_id DESC` to
  `last_active_at DESC NULLS LAST, created_at DESC, account_id DESC`. Profiles
  whose account has never been active (`NULL`) sort last. **BREAKING** only for
  a client that asserts strict chronological deck order; the response **schema
  is unchanged** (same fields, opaque `next_cursor`).
- **NEW** deck-specific keyset cursor (`encodeDeckCursor` / `decodeDeckCursor`)
  carrying the 3-tuple `(last_active_at, created_at, account_id)` with correct
  `NULLS LAST` boundary handling. The shared 2-tuple `encodeCursor` /
  `decodeCursor` (used by status, social, messaging) is **left untouched**.
  Cursors issued in the old 2-tuple format are treated as invalid by the deck
  decoder and the request restarts from the top of the deck (self-healing across
  the deploy boundary).
- **SCOPE**: ordering + activity tracking only. No compatibility/affection
  scoring, no account-status filtering of the deck, no denormalization onto
  `dating_profiles`, no change to swipe/match logic. `last_active_at` is exposed
  on no API response in this change.

## Capabilities

### New Capabilities

- `last-active-tracking`: the `accounts.last_active_at` column and the
  debounced, non-blocking per-request write in the auth resolver that keeps it
  fresh.
- `dating-deck-ordering`: the recency-of-activity deck sort, the 3-key ordering
  with `NULLS LAST`, and the deck keyset cursor with its boundary semantics.

## Impact

**Affected packages and apps:**

- `packages/db` — add `lastActiveAt` to the `accounts` table; generated
  migration.
- `apps/web` — the auth resolver (activity write), the dating deck handler
  (join + reorder + new cursor), and the pagination lib (deck cursor helpers).

**Files created or modified:**

- MODIFY `packages/db/src/schema/accounts.ts` — add
  `lastActiveAt: date('last_active_at', { mode: 'string' })` (nullable).
- CREATE `packages/db/migrations/<generated>_accounts_last_active_at.sql`
  (generated by `mise run db:generate`).
- MODIFY `apps/web/src/auth/macro.ts` — fire the guarded, non-blocking activity
  UPDATE inside `resolve` after the bearer verifies.
- MODIFY `apps/web/src/modules/dating/index.ts` — rewrite the deck query from
  the relational builder (`db.query.datingProfiles.findMany`) to a core
  `db.select(...).innerJoin(accounts, ...)` so it can `ORDER BY` a joined
  column; new ordering, new keyset predicate, new deck cursor.
- MODIFY `apps/web/src/lib/pagination.ts` — add `encodeDeckCursor` /
  `decodeDeckCursor`; the existing helpers are unchanged.
- MODIFY `apps/web/src/__tests__/pagination.test.ts` — cover the deck cursor
  round-trip, `NULLS LAST`/null `last_active_at`, and old-format rejection.
- MODIFY/CREATE dating + auth-resolver tests covering the new ordering and the
  debounced write (`apps/web/src/modules/dating/*.test.ts`,
  `apps/web/src/auth/*.test.ts`).

**Backward compatibility:** Mixed.

- Schema: **additive** — a nullable `date` column; Postgres adds it without a
  table rewrite; no backfill (NULL means never-active, which sorts last).
- API: response **schema unchanged**; `next_cursor` stays opaque. Deck
  **ordering semantics change** (behavioral) — flagged **BREAKING** only for
  clients pinning chronological order. Old-format cursors self-heal by
  restarting at page 1.
- No endpoint, column, field, or CLI command is removed or renamed.
